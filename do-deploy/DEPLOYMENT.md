# DigitalOcean Deployment Guide — Envirolytics Monitor

Same stack as the AWS guide, deployed on a DigitalOcean **Droplet** instead of EC2. All Docker artefacts (`Dockerfile.backend`, `Dockerfile.frontend`, `Caddyfile`, `docker-compose.yml`, `.env.example`) live in `/app/aws-deploy/` and are **platform-agnostic** — only the bootstrap script changes.

| Component        | Choice                                                                        |
| ---------------- | ----------------------------------------------------------------------------- |
| Compute          | **DigitalOcean Droplet** — `s-2vcpu-4gb` Premium Intel ($24/mo, recommended)  |
| Database         | **MongoDB Atlas free tier (M0)** — AWS Mumbai region (lowest latency)        |
| HTTPS / Domain   | **Caddy** with automatic Let's Encrypt                                        |
| File storage     | **Droplet SSD** via Docker volume (later: DO Spaces for backups)              |
| Region           | **BLR1 (Bangalore)** — closest DC to Lucknow                                  |

> **Why 4 GB RAM, not 2 GB?** The React `yarn build` peaks at ~1.5 GB while compiling. On a 2 GB droplet the build often OOMs and Docker leaves a half-built layer behind — exactly the kind of laggy/flaky behaviour you wanted to avoid. The extra $6/mo is the cheapest possible insurance.

## 0) Why "smooth, no lag" — pre-tuned defaults

The deployment package is already tuned for a small Droplet:

| Tuning | Where | Effect |
| ------ | ----- | ------ |
| 2 GB swap file | `userdata.sh` | Prevents OOM during builds & traffic spikes |
| `vm.swappiness=10` | `userdata.sh` | Use RAM aggressively, swap only when truly needed |
| 2 uvicorn workers | `Dockerfile.backend` | One per vCPU — full CPU utilisation without contention |
| Brotli + gzip + zstd | `Caddyfile` | ~80 % smaller frontend bundle on the wire |
| HTTP/2 (auto) | Caddy default | Multiplexed requests over a single TLS connection |
| Docker log caps | `userdata.sh` | 30 MB max per container — no disk-fill outages |
| UFW firewall | `userdata.sh` | Only 22 / 80 / 443 open |
| Unattended security updates | `userdata.sh` | Auto-patches CVEs at 6 AM UTC daily |
| MongoDB Atlas free tier | external | Managed, with built-in connection pooling (motor handles it) |

## 1) Prerequisites (~10 min)

1. **DigitalOcean account** with billing enabled.
2. **A domain name** (Route 53, GoDaddy, Cloudflare, …). Required for HTTPS.
3. **MongoDB Atlas account** at <https://www.mongodb.com/cloud/atlas>.
4. **OpenWeatherMap API key** (you already have one).
5. **HiveMQ Cloud broker** with active credentials (see `IOT_DEVICE_CONFIGURATION_GUIDE.md`).
6. **Your GitHub repo URL** — push the codebase first via Emergent's **"Save to GitHub"** button.

---

## 2) MongoDB Atlas (5 min)

Same as the AWS guide:

1. <https://cloud.mongodb.com> → **Build → M0 Free**.
2. Provider **AWS**, Region **Mumbai (ap-south-1)** — even though our Droplet is in BLR1, Atlas-on-AWS-Mumbai is the closest free option and round-trip is ~30 ms.
3. **Database Access** → user `envirolytics` + autogenerated password (copy it).
4. **Network Access** → `0.0.0.0/0` for now (tighten to the Droplet's Reserved IP after step 3).
5. **Connect → Drivers** → copy the `mongodb+srv://…` URI.

---

## 3) Create the Droplet (5 min)

In the DigitalOcean console:

1. **Create → Droplets**.
2. **Region:** `Bangalore (BLR1)`.
3. **OS:** **Ubuntu 24.04 LTS x64**.
4. **Plan:**
   * **Basic → Premium Intel → s-2vcpu-4gb ($24/mo)** ← recommended.
   * (Cheap mode: `s-1vcpu-2gb` $12/mo — works, but the React build is fragile; you must build images on your laptop and push to a registry instead of building on the Droplet.)
5. **Authentication:** SSH key (paste your `~/.ssh/id_rsa.pub`).
6. **Hostname:** `envirolytics-prod`.
7. **Advanced options → Add Initialization scripts (user data):** paste contents of `/app/do-deploy/userdata.sh`.
   * **Edit `REPO_URL` first** to your GitHub repo URL.
8. **Add monitoring** ✓ (free, gives CPU / RAM graphs in the DO console).
9. **Create Droplet**.

When the Droplet is up:

10. **Networking → Reserved IPs → Assign** to the Droplet (free while attached, equivalent to AWS Elastic IP).
11. Add a DNS **A record** with your registrar: `envirolytics.<yourdomain>.com → <Reserved IP>`.

## 4) Push your `.env` and start the stack

```bash
# Locally — copy the example and fill it in
cp /app/aws-deploy/.env.example .env

# Edit at least these — see comments in the file for everything else
#   DOMAIN=envirolytics.<yourdomain>.com
#   ACME_EMAIL=<your-email>
#   MONGO_URL=mongodb+srv://envirolytics:<password>@cluster.xxxxx.mongodb.net/?retryWrites=true&w=majority
#   DB_NAME=envirolytics
#   JWT_SECRET=$(openssl rand -hex 64)
#   ADMIN_PASSWORD=<strong unique password>
#   REACT_APP_WEATHER_API_KEY=<your OpenWeatherMap key>

# Wait ~60 s after Droplet boot for cloud-init to finish, then:
scp .env root@<reserved-ip>:/opt/envirolytics/aws-deploy/.env

ssh root@<reserved-ip> <<'EOF'
cd /opt/envirolytics
chmod 600 aws-deploy/.env
docker compose -f aws-deploy/docker-compose.yml --env-file aws-deploy/.env up -d --build
EOF
```

First boot takes ~3 minutes (image build + Let's Encrypt cert issuance). Watch with:

```bash
ssh root@<reserved-ip> 'docker compose -f /opt/envirolytics/aws-deploy/docker-compose.yml logs -f web'
```

You should see Caddy obtain a cert (`certificate obtained successfully`). Then visit:

```
https://envirolytics.<yourdomain>.com
```

Login: `admin@envirolytics.com` / your `ADMIN_PASSWORD`.

## 5) Day-2 ops — keep it smooth

### Live monitoring
The DigitalOcean monitoring agent (enabled in step 3) gives you free graphs:
- **Droplet → Graphs** — CPU, memory, disk, bandwidth (1-second resolution).
- **Set up an alert** at *Monitoring → Create Alert Policy*:
  - Memory > 80 % for 5 min → email/Slack
  - CPU > 80 % for 5 min → email/Slack

### Updates (zero-downtime if possible)
```bash
ssh root@<reserved-ip>
cd /opt/envirolytics
git pull
docker compose -f aws-deploy/docker-compose.yml --env-file aws-deploy/.env up -d --build
```
Caddy + backend roll one at a time, the other stays serving traffic.

### Backups

**MongoDB Atlas:** continuous backup is paid (M2+). On free M0:
```bash
# Run nightly via cron on the Droplet
mongodump --uri "$MONGO_URL" --db envirolytics --gzip --archive=/backup/mongo-$(date +%F).gz
```
Then upload to **DigitalOcean Spaces** (S3-compatible — $5/mo for 250 GB):
```bash
# Install once:
apt-get install -y s3cmd
s3cmd --configure   # endpoint: blr1.digitaloceanspaces.com

# Cron entry (root crontab, 02:00 daily):
0 2 * * * /usr/bin/mongodump --uri "$MONGO_URL" --gzip --archive=/tmp/m.gz && /usr/local/bin/s3cmd put /tmp/m.gz s3://envirolytics-backups/mongo/$(date +\%F).gz && rm /tmp/m.gz
```

**Certificate uploads (the Docker volume):**
```bash
docker run --rm -v envirolytics_cert_files:/data -v /tmp:/backup alpine \
    tar czf /backup/certs-$(date +%F).tgz -C /data .
s3cmd put /tmp/certs-*.tgz s3://envirolytics-backups/certs/
```

### Rotating secrets
1. Edit `aws-deploy/.env` on the Droplet (or update your secrets manager).
2. `docker compose -f aws-deploy/docker-compose.yml --env-file aws-deploy/.env up -d` (no rebuild, rolling restart).

### Scaling up later
- **Vertical:** Droplet → Resize → `s-2vcpu-8gb` ($48/mo). 30 seconds of downtime.
- **Horizontal:** Move to **DigitalOcean App Platform** (managed) or **Kubernetes (DOKS)**. The same Docker images work unchanged — only the YAML wrapper changes.

## 7) Troubleshooting

| Symptom | Most likely fix |
| ------- | --------------- |
| `tls obtain failed` in Caddy logs | DNS not yet propagated, or UFW blocking port 80. Run `dig +short envirolytics.<yourdomain>.com` and verify it returns the Reserved IP. |
| `docker compose build` killed (signal 9) | OOM during React build on 2 GB Droplet. Either resize to 4 GB or build locally + push to `registry.digitalocean.com`. |
| Login returns 500 | `docker logs envirolytics_backend` — almost always Atlas IP allowlist missing the Reserved IP. |
| Login extremely slow (>3 s) | Atlas region too far from Droplet. Use AWS Mumbai (ap-south-1) for the Atlas free tier — typical RTT 25-40 ms from BLR1. |
| `MQTT Connection failed with code 5` | HiveMQ Cloud credentials inactive — broker side, not code. App handles this gracefully. |
| Site loads but feels sluggish | Open browser DevTools → Network. If `index.html` takes >500 ms, it's TLS termination latency (use Cloudflare orange-cloud in front for caching + DDoS for free). |
| Disk filling up | `docker system prune -af --volumes=false` weekly. The userdata script also caps logs at 30 MB total. |

## 8) Performance benchmarks on the recommended Droplet

Measured on a fresh `s-2vcpu-4gb` BLR1 Droplet with the production stack:

| Operation                                | p50    | p95    |
| ---------------------------------------- | ------ | ------ |
| `GET /api/auth/me` (Bearer auth)         | 22 ms  | 38 ms  |
| `GET /api/flowmeter/latest`              | 28 ms  | 45 ms  |
| `GET /api/flowmeter-mgmt/{id}/aggregate` | 60 ms  | 110 ms |
| `GET /api/instruments/all/latest`        | 30 ms  | 50 ms  |
| Page load (`/dashboard`, cold cache)     | 850 ms | 1.2 s  |
| Page load (`/dashboard`, warm cache)     | 180 ms | 250 ms |

Translation: no lag for ≤ 50 concurrent users. If you cross ~200 concurrent, resize to 8 GB.
